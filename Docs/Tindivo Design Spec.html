00Resumen del producto

Tindivo es una plataforma de delivery y gestión de pedidos para restaurantes pequeños y medianos en San Jacinto, Áncash. Su objetivo en el piloto es migrar los pedidos que hoy ocurren por WhatsApp a una experiencia web estructurada con seguimiento en tiempo real.

Restaurantes en el catálogo

RestauranteEstadoTipoNotas
PriamoActivo (piloto)Pizzas & hamburguesasÚnico con flujo completo, badge "Más pedido".
La NonnaPróximamentePastas frescasCard visible en 40% opacidad, sin metadata, no clickeable.
La FlorenciaPróximamenteComida criollaMismo estado inactivo.

Stack referencial

  • Frontend: Next.js (PWA), React
  • Backend: Supabase (Postgres + Realtime + Auth + Storage)
  • Mapas: Google Maps JavaScript API con marcador draggable
  • Push: Web Push API (sólo panel del restaurante)
  • Pago: directo a Yape del restaurante (Tindivo no retiene fondos)

01Sistema de diseño

1.1 Identidad

Tindivo proyecta cercanía y velocidad en zona rural. La paleta gira en torno al naranja Tailwind (acogedor, comestible, con energía) sobre un crema cálido. La tipografía display es generosa y bold; el cuerpo es neutro y legible. La iconografía es lineal, 2px stroke, no decorativa. Las fotos de comida son protagonistas; los placeholders se evitan en cards de restaurante.

1.4 Espacio y radios

  • Grilla mental: 8pt. Padding interno de cards 14–16. Padding lateral de pantalla 16–20. Padding superior debajo del status bar 54px.
  • Radios: 10 (chips internos), 12 (chips/badges), 14 (banners callout), 16 (inputs/fields), 18 (botones primarios, address cards), 22 (cards y banners principales), 28 (esquina superior del bottom sheet), 999 (pills, avatars, dots).
  • Sombras de marca: 0 8px 22px -8px rgba(249,115,22,0.55) para botones primarios; sombras suaves de elevación 0 1px 2px rgba(0,0,0,0.04) para cards interactivas.

1.5 Iconografía

Conjunto inline SVG propio: Back, Close, Plus, Minus, Check, Search, Bag, Star (no usado en el flujo final), Clock, Pin, Upload, Phone, Truck, Store. Stroke 2 – 2.4px, esquinas redondeadas. Iconos sobre fondo dark blur se invierten a blanco (header de menu).

1.6 Imagery

  • Cards de restaurante: foto cuadrada real (88×88, radius 14), background-size: cover.
  • Hero de restaurante: foto fullbleed 300px de alto con gradient overlay linear-gradient(180deg, rgba(0,0,0,0.15) 0%, transparent 35%, rgba(0,0,0,0.75) 100%). Texto blanco con text-shadow.
  • Productos del menú: placeholder rayado (45°) con etiqueta mono. Se sustituye por fotos reales cuando estén disponibles.

04Flujos de usuario

4.1 Primer pedido (sin sesión)

Landing→ Menu Priamo→ Product Modal→ Cart→ Onboarding sheet→ Checkout→ Confirm modal→ Payment method→ Prepay upload (si aplica)→ Order Confirmed→ Tracking

El onboarding sheet se inserta en el cart→checkout cuando el usuario no está logueado. Empieza en method y avanza hasta address; al terminar, aterriza directo en checkout con la dirección recién creada como default.

4.2 Usuario recurrente

Misma cadena pero salta el onboarding (ya tiene phone + addresses). Se reabre solo si el usuario logueado eliminó su dirección por defecto o no tiene teléfono.

4.3 Dismissal y reapertura del onboarding

  • X en method: cierra el sheet, usuario sigue no logueado.
  • X en phone o address: cierra el sheet, mantiene la cuenta parcial (name+email solo).
  • En email la X se convierte en Back y vuelve a method.
  • Reapertura automática al intentar avanzar de cart a checkout: si !isOnboardingComplete(user), se vuelve a abrir el sheet en el sub-paso correcto.

4.4 Timeout de pago

Prepay Upload (timer 10:00)→ 00:00 sin upload→ Order Cancelled (timeout)→ Volver al menú

Aplica idéntico para Pick-up (siempre) y Delivery + Prepagar por Yape (opcional).

4.5 Cancelación manual

Tracking · estado sent→ Tap "Cancelar pedido"→ Confirm dialog→ Order Cancelled (user)

Solo permitido en estado sent. En confirmed y posteriores el botón se sustituye por mensaje informativo + SupportLink.

4.6 Soporte WhatsApp

El componente SupportLink aparece en Order Confirmed, Tracking, Order Cancelled y en el footer del estado no-cancelable. Abre wa.me/51987654321 con mensaje pre-rellenado "Hola Tindivo 👋, tengo un problema con mi pedido #TND-XXXXX."

5.5 Restaurantes

  • Solo Priamo está activo en el piloto. Las cards inactivas se muestran al 40% opacidad + grayscale 100%, sin metadata extra, sin badge "Próximamente" textual.
  • El menú se organiza en secciones: Pizzas, Hamburguesas, Bebidas.
  • Cada producto tiene categoría que dicta sus modifier groups (Pizzas: Tamaño + Masa + Extras; Burgers: Cocción + Acompañamiento + Extras + Salsa; Bebidas: ninguno).

5.6 Constraints del piloto

Decisiones de simplicidad para validar antes de escalar:
  • Sin verificación OTP del teléfono
  • Sin verificación de email
  • Sin pasarela de pago integrada (solo Yape directo)
  • Sin app nativa — todo es PWA
  • Sin tracking GPS del motorizado en mapa
  • Sin modificadores negativos estructurados (se resuelve con nota libre)
  • Carta editada por admin Tindivo, no por el restaurante

5.7 Cobertura geográfica

Polígono dibujado sobre el mapa de San Jacinto. Mapa centrado por defecto en -9.1547, -78.5042. Las faldas del cerro están excluidas por seguridad del motorizado. Cliente fuera de zona ve: "Lo sentimos, tu ubicación está fuera de nuestra zona de cobertura actual. Puedes hacer un pedido de Pick-up directamente en el local."

7.2 Estado clave de la app

  • screen: landing | menu | cart | checkout | payment | prepay | confirmed | tracking | cancelled | account | addressEdit
  • user: { signedIn, name, email, phone, addresses: Address[] }
  • order: { method: 'delivery' | 'pickup', addressId, phone, note }
  • submittedOrder: { id, phone, method, payment: { methodId, methodLabel, change } }
  • trackingState: id de TRACKING_STATES
  • cancelReason: 'timeout' | 'user'
  • authOpen: boolean para mostrar el bottom sheet de onboarding

7.3 Gates principales

  • isOnboardingComplete(user): user.signedIn && user.phone && user.addresses.length > 0
  • handleOpenCart: si no signed in, abre sheet + navega a cart; si signed pero incompleto, abre sheet + navega; si completo, va directo.
  • handleProceedToCheckout: si no completo, reabre sheet; si completo, navega a checkout.

7.5 Producción — sustituciones recomendadas

  • Mapa: reemplazar MapView stylized por Google Maps JS API con marcador draggable y polígono de cobertura. Mantener overlays UI (chip de coords, hint, botón mi ubicación).
  • Auth: integrar Supabase Auth (Google OAuth + email) preservando los sub-pasos del bottom sheet como están.
  • Upload de comprobante: usar Supabase Storage. La UI dropzone se mantiene.
  • Realtime: suscribir el tracking a Supabase Realtime para que los cambios de estado vengan del backend; mantener el timeline UI como vive.
  • Push: solo para el panel del restaurante (no este producto). Cliente recibe updates vía la pantalla de tracking.
